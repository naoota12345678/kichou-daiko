# Part 1: プロダクト概要

## 1.1 ビジョン

「社長はレシートを撮って、ボタンを押すだけ。あとは全部自動。」

あらゆるWebサービスの取引データと紙のレシートを、社長自身のGoogleドライブに
電子帳簿保存法準拠の形式で自動保存し、同時に仕訳CSVを生成するアプリ。
単体で完結する電帳法対応ツールとして機能しつつ、
プロ連携版では経営コンサルタントのAI判定付き記帳代行サービスにつながる。

## 1.2 製品の本質

このアプリは「会計ソフト」ではない。
**「あらゆるWebサービスの取引データをGoogleドライブに集約し、仕訳CSVを自動生成するアプリ」。**

どの会計ソフトとも喧嘩しない。freeeでもMFクラウドでも弥生でも財務応援R4でも、
このアプリが作ったCSVをインポートするだけ。全ての会計ソフトの「手前」に入るツール。

## 1.3 解決する問題

社長が毎月抱えている3つの面倒を同時に消す：

**問題①** Amazonで何買ったか調べて仕訳にしなきゃいけない
→ freeeもMFクラウドもカード明細に「AMAZON.CO.JP ¥12,500」としか表示しない
→ 品目がわからないから科目が決められない
→ **このアプリ：Amazon/楽天にログインして品名まで自動取得 → 仕訳CSV自動生成**

**問題②** Amazon/楽天の領収書を電子保存しなきゃいけない（電帳法）
→ 2024年1月から全事業者に義務化済み。違反は青色申告取消リスク
→ でもほとんどの社長が対応できていない
→ **このアプリ：領収書PDFを電帳法準拠のファイル名でGoogleドライブに自動保存**

**問題③** 紙のレシートを税理士に渡さなきゃいけない
→ 紙を郵送、または写真をメールで送る手間
→ **このアプリ：動画で撮影 → 画像をGoogleドライブに保存 → 税理士と共有**

## 1.4 freeeとの差別化

| 観点 | freee | このアプリ |
|------|-------|-----------|
| Amazon品目特定 | × カード名義のみ | ◎ 品名・数量まで取得 |
| 楽天品目特定 | × カード名義のみ | ◎ 品名まで取得 |
| レシート取込 | 1枚ずつ撮影 | 動画で一括取込 |
| 電帳法保存先 | freeeのサーバー | 社長自身のGoogleドライブ |
| データ所有権 | freee側 | 社長本人 |
| 月額 | 2,680円〜 | 500円 |
| 仕訳CSV | freee専用 | 全会計ソフト対応 |
| 位置づけ | 会計ソフト（競合する） | 補完ツール（共存する） |

**freeeユーザーにとっても「freee＋このアプリ」で初めてAmazon問題が解決する。**

# Part 2: アプリ設計

## 2.1 ホーム画面

```
[ログイン画面]
  → 顧客ID + PINコード

[ホーム画面]

  📷 レシートを撮る
     → 動画撮影モード

  🛒 取り込む
     → Amazon
     → 楽天
     → MFクラウド
     → freee
     → モノタロウ
     → カード明細（楽天カード / 三井住友 / ...）
     → ネットバンク（○○銀行 / ...）
     → ＋ サイトを追加

  📂 保存済みの領収書を見る
     → 日付・金額・取引先で検索可能（電帳法の検索要件を満たす）

  📊 今月の状況
     → レシート 15枚撮影済み
     → Amazon 28件取り込み済み
     → 楽天 12件取り込み済み
     → 仕訳CSV 出力可能
```

## 2.2 初回設定フロー

```
① 顧客IDでログイン

② 「領収書の保存先を設定してください」
   → Googleアカウントでログイン
   → Googleドライブのフォルダを選択
   → アプリが書き込み権限を取得（drive.fileスコープ）
   → 社長の他のファイルには一切アクセスしない

③ お使いの会計ソフトを選択（CSV出力形式の設定）
   → freee / MFクラウド / 弥生 / 財務応援R4 / その他

④ 決算月を選択（フォルダの年度区切り用）
   → 3月決算 / 12月決算 / その他

⑤ 事務処理規程の確認
   → アプリが電帳法の事務処理規程を自動生成
   → 「同意する」をタップ
   → 規程PDFがGoogleドライブに自動保存

⑥ 完了
```

## 2.3 取り込みの仕組み（WebView方式）

```
社長が「Amazonの明細を取り込む」を押す
  → アプリ内でAmazonの注文履歴ページを開く（WebView）
  → 社長がID/パスワードでログイン（Amazonに直接入力）
  → 注文履歴ページが表示される
  → 「取り込み」ボタンを押す
  → アプリがHTMLを解析して品名・金額・日付を抽出
  → 同時に領収書ページからPDFを生成
  → すべてGoogleドライブに自動保存
  → 「3月分 28件取り込みました ✓」
```

パスワードはアプリに保存しない。毎回社長が自分でログインする。月1回だから苦にならない。
楽天もMFクラウドもfreeeもカード会社も銀行も全部同じ仕組み。

## 2.4 レシート撮影

```
社長が「レシートを撮る」を押す
  → 動画撮影モードが起動
  → レシートを机に並べてスマホでバーッと撮影
  → 撮影完了 → アップロード

[単体アプリ版]
  → 動画からベストフレームを抽出
  → 1枚ずつの画像に分割
  → ファイル名は連番で仮設定：20260310_レシート_001.jpg
  → Googleドライブに保存
  → OCRは行わない（AIコストゼロ）

[プロ連携版]
  → 上記に加えてClaude Vision OCRで品目・金額を抽出
  → 電帳法ファイル名に自動リネーム：20260310_セブンイレブン_1080.jpg
  → 仕訳データに変換
```

## 2.5 通知機能

```
月末（例：25日）にアプリからプッシュ通知：

  「〇〇社長、今月の領収書を取り込んでください。
   レシート撮影とAmazon/楽天の取り込みをお願いします。」

社長が取り込み完了後 → 尚子さん側に通知：
  「〇〇社 3月分の資料が届きました」

→ 催促の電話・メールが不要になる
```

# Part 3: Googleドライブ設計

## 3.1 フォルダ構成

```
📁 社長が指定したフォルダ/
  📁 事務処理規程/                    ← 電帳法の規程PDF
    📄 電子取引データ保存に関する事務処理規程.pdf
  📁 2025年度/
    📁 04月/
      📁 領収書/                      ← 原本保存ゾーン（書き換え不可）
        📄 20250405_Amazon_USBハブ_2500_T9876543210987.pdf
        📄 20250410_セブンイレブン札幌北口_1080.jpg
      📁 仕訳データ/                  ← CSV出力先
        📄 20250430_Amazon_仕訳.csv
        📄 20250430_楽天_仕訳.csv
        📄 20250430_レシート_仕訳.csv  ← プロ連携版のみ
    📁 05月/
    ...
  📁 2026年度/
    📁 04月/
    ...
```

## 3.2 領収書フォルダとCSVフォルダを分ける理由

**領収書フォルダ（原本ゾーン）：**
- 電帳法の原本保存場所。書き換え・削除禁止
- アプリからの書き込み専用。上書き・削除はアプリ経由でもログ記録必須
- ファイル名に日付・取引先・金額・インボイス番号を含む
- 7年間保存
- 尚子さんが取り込む必要なし。見るだけ。

**仕訳データフォルダ（CSV出力先）：**
- 会計ソフトにインポートするための加工データ
- 修正・再出力が可能
- 尚子さんがここからCSVを取得して処理

## 3.3 ファイル命名規則

```
[EC購入の領収書]
  {YYYYMMDD}_{取引先}_{品名}_{金額}_{インボイス番号}.pdf
  例：20260305_Amazon_コピー用紙A4_3980_T1234567890123.pdf

[レシート撮影画像]
  単体版：{YYYYMMDD}_レシート_{連番}.jpg
  プロ連携版：{YYYYMMDD}_{店名}_{金額}.jpg

[仕訳CSV]
  {YYYYMMDD}_{ソース}_仕訳.csv
  例：20260331_Amazon_仕訳.csv
```

## 3.4 ストレージ容量

```
月50件取り込む社長の場合：
  PDF 30枚 × 200KB = 6MB
  画像 20枚 × 2MB  = 40MB
  月合計 ≒ 50MB
  年間 ≒ 600MB
  7年間 ≒ 4.2GB

Googleドライブ無料枠 = 15GB → 十分余裕あり
年度でドライブを変える必要なし
```

## 3.5 権限設計

```
Google Drive API スコープ：drive.file
→ アプリが自分で作成したファイルにのみアクセス可能
→ 社長のドライブの他のファイルには一切触れない
→ セキュリティとプライバシーを担保

共有設定：
→ 社長が任意で税理士のメールアドレスを追加可能
→ 税理士はブラウザでドライブを開くだけで全領収書を参照可能
→ 尚子さんでも、別の税理士でもOK
```

## 3.6 重複チェック

```
新しいデータが来るたびに：

① ドライブの既存ファイル名を検索
   → 日付 × 金額 × 取引先が一致 → 重複候補

② 重複パターンの検出：
   A. レシート撮影 × Amazon取り込み → 二重計上防止
   B. カード明細 × レシート撮影 → 統合
   C. 同じレシートの二重撮影 → 自動排除

③ 重複検出時：
   → 「このレシートは取り込み済みです」と通知
   → 自動で弾く or 社長に確認
```

# Part 4: 仕訳CSV自動生成

## 4.1 Amazon/楽天の仕訳（AIコストゼロ）

WebViewで取得したHTML内の構造化データから、プログラム的に仕訳CSVを生成。
AIもOCRも不要。

```
[取得できるデータ]
  日付：2026/03/08
  品名：USBハブ 4ポート
  金額：¥2,500
  取引先：Amazon
  インボイス番号：T9876543210987
  支払方法：Visa末尾1234

[科目推定ルール（キーワードマッチング）]
  品名に「書籍」「本」→ 新聞図書費
  品名に「インク」「トナー」「用紙」→ 消耗品費
  品名に「USB」「ケーブル」「マウス」→ 消耗品費
  品名に「文房具」「ペン」「ノート」→ 事務用品費
  金額10万円以上 → 要確認（資産計上の可能性）
  上記に該当しない → 「不明」として出力

[生成されるCSV]
  日付,借方科目,借方金額,貸方科目,貸方金額,摘要
  2026/03/05,消耗品費,3980,未払費用,3980,Amazon コピー用紙A4
  2026/03/08,消耗品費,2500,未払費用,2500,Amazon USBハブ
  2026/03/08,新聞図書費,1200,未払費用,1200,Amazon 経営の教科書
  2026/03/08,消耗品費,8800,未払費用,8800,Amazon インクカートリッジ
```

貸方は全部「未払費用」で統一。freeeでもMFでも弥生でもインポートすれば振り替わる。

## 4.2 会計ソフト別CSV出力

初回設定で選択した会計ソフトに合わせてCSV形式を自動切替：

```
├── freee形式CSV（振替伝票インポート用）
├── MFクラウド形式CSV
├── 弥生形式CSV
├── 財務応援R4形式CSV（Shift_JIS、摘要48Byte制限対応）
└── 汎用CSV（日付・借方科目・貸方科目・金額・摘要）
```

## 4.3 レシート分の仕訳

```
[単体アプリ版]
  レシートは画像保存のみ。仕訳CSVは出さない。
  社長が自分で会計ソフトに入力 or 税理士が対応。

[プロ連携版]
  Claude Vision OCRで品目・金額・日付・インボイス番号を抽出
  → AI判定 → 仕訳CSV生成
  → 尚子さんの注意レポートで確認
```

# Part 5: 電子帳簿保存法対応

## 5.1 対応方針

2024年1月より電子取引で受領した領収書等は電子データのまま保存が義務。
紙に印刷して保存は不可。

**タイムスタンプは不要。** 事務処理規程の策定・運用で真実性を担保する。
国税庁がサンプルを公開しており、アプリが初回設定時に自動生成する。

売上高1,000万円以下の小規模事業者は、税務職員のダウンロード要求に応じられれば
検索要件すら全て不要。

## 5.2 このアプリが満たす要件

```
[真実性の確保]
  → 事務処理規程を自動生成しGoogleドライブに保存
  → アプリの操作ログで「誰が・いつ・何を保存/削除したか」を記録
  → タイムスタンプ不要

[可視性の確保]
  → PC + ディスプレイ + プリンタ → 社長の事務所にあるはず
  → 検索機能 → ファイル名に日付・取引先・金額を含むため
    Googleドライブの検索機能で検索可能
  → アプリ内の「保存済みの領収書を見る」からも検索可能

[保存期間]
  → Googleドライブに7年間保存
  → ドライブ容量は十分（7年で約4GB）

[インボイス対応]
  → ファイル名末尾にインボイス番号（T+13桁）を含む
  → インボイス番号での検索が可能
```

## 5.3 競合の電帳法対応ツールとの比較

```
バクラク、TOKIUM等（月数千円〜数万円）：
  提供するもの：タイムスタンプ自動付与、JIIMA認証、ワークフロー承認
  → 小規模事業者には不要な機能に高い月額を払っている

このアプリ（月500円）：
  提供するもの：ファイル名自動付与、事務処理規程自動生成、Googleドライブ保存
  → 法令準拠に必要十分な機能のみ。過不足なし。
```

# Part 6: AI判定エンジン（プロ連携版）

## 6.1 設計思想

**「判定 → 確認 → 仕訳」の順番。**
仕訳を作ってからチェックするのではなく、判定と確認が終わってから仕訳を生成する。
出てきた仕訳＝完成品。もう見直す必要がない。

## 6.2 3層判定アーキテクチャ

### 第1層：パターンマッチ（過去実績ベース）
```
過去3ヶ月以内に同一取引先＋同一金額帯＋同一科目の実績がある
→ 確信度0.90以上 → 白（自動承認）
→ 全体の40〜50%を占める定型取引
```

### 第2層：ルールエンジン（業種知識ベース）

**飲食の判定：**
- 1人分1,000円以下 → 会議費
- 1人分5,000円以下 → 会議費
- 1人分5,000円超 → 交際費
- 休日の飲食 → 黄色フラグ
- 居酒屋・バー → 交際費

**EC購入の判定（品目特定後）：**
- 書籍（業務関連）→ 新聞図書費
- 漫画 → 黄色「福利厚生費（待合室用）の可能性」
- 事務用品10万円未満 → 消耗品費
- 10万円以上 → 黄色「資産計上の判断が必要」

**宿泊・交通の判定：**
- 勤務地近辺 → 旅費交通費
- 観光地 → 黄色「出張 or 私的旅行の確認。按分提案」
- 休日のフライト → 黄色

**業種別プロファイル：**
同じ支出でも業種で判断が変わる。
- 経営コンサルタント → ホテル代は「サービス品質調査」で経費性高い
- 建設業 → ホームセンターは「材料・工具」で経費性高い
- 美容業 → 化粧品は「施術用材料」、雑誌は「待合室用」
- 飲食業 → 他店での飲食は「競合調査」

### 第3層：Claude AI総合判断

第1層・第2層で判定できない場合、クライアントの業種・事業内容・過去の処理パターンを
文脈として渡してClaudeに総合判断させる。

## 6.3 按分処理

```
AI「宿泊費 ¥85,000。観光地。家族同伴の可能性あり。」

注意レポート：
  「宿泊人数は？」
  → [1人] → 全額経費
  → [2人] → ¥42,500を経費 / ¥42,500を事業主貸
  → [4人] → ¥21,250を経費 / ¥63,750を事業主貸
  
尚子さんが選択 → 仕訳が自動分割される
```

## 6.4 注意レポート

仕訳一覧を全件スクロールする方式は採用しない。
白（自動承認）はそのまま流れ、黄・赤だけが別ビューで表示される。

```
■ 経費性の確認（3件）

  03/05 Amazon 漫画3冊 ¥1,980 → 福利厚生費
  理由：待合室・休憩室用であれば経費計上可。
  → [はい] [いいえ]

  03/22 ニセコ温泉旅館 ¥85,000 → ???
  理由：観光地の宿泊。
  → [全額経費] [按分] [経費除外]

■ 品目不明のカード明細（1件）

  03/15 楽天カード ¥5,400 → 品目不明
  → [品目を入力] [保留]

■ サマリ
  白（自動承認）：142件
  黄（AI判断済）：5件
  赤（要確認）  ：1件
```

## 6.5 安全弁（強制フラグ）

AIの確信度に関わらず必ず人間確認に回すもの：
- 10万円超の単発支出
- 交際費（全件）
- 休日の支出
- 初めての取引先
- 観光地の宿泊
- 品目不明のカード明細（1万円超）

## 6.6 学習サイクル

尚子さんの修正がルールにフィードバックされる。
3月に「漫画→福利厚生費」で承認 → 4月の同じ顧問先の漫画は白で通る。
四半期ごとに自動承認に昇格した項目を一覧で見直す安全弁あり。

## 6.7 判定記録（税務調査対応）

全ての判断について根拠を構造化して記録。
「なぜこれを経費にしたか」が後から説明できる状態を維持。

# Part 7: カード明細×EC購買 突合エンジン（プロ連携版）

## 7.1 問題

カード明細には「AMAZON.CO.JP ¥12,500」としか表示されない。
freeeを含む既存の会計ソフトはこの問題を解決できていない。

## 7.2 解決

カード明細（Moneytree）とEC購買情報（アプリで取り込み済み）を突合。
アプリがAmazon/楽天の品目データを持っているから、カード明細に品目を補完できる。

```
[Moneytree]  03/08 AMAZON.CO.JP ¥12,500
[アプリ]     03/07 USBハブ ¥2,500 + 書籍 ¥1,200 + インク ¥8,800 = ¥12,500

→ 突合成功。カード明細が3行の仕訳に分解される。
```

## 7.3 突合ロジック

- 突合キー：日付（±3日）× 金額（完全一致 or 合計一致）
- 1対1マッチ：カード1件 = 注文1件
- 1対Nマッチ：カード1件 = 注文N件（合計金額一致）
- マッチなし：品目不明 → 黄色フラグ

## 7.4 二重計上防止

レシートで撮影した支出がカードで払われていた場合、
Moneytreeのカード明細とレシートOCR結果を照合して二重計上を検出。
